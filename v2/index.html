Polymarket 与预测市场深度分析报告：校订版

从事实核查、市场结构、监管风险到 AgentPort 产品可行性的重新评估

版本：2026-06-30 · 本文为公开资料整理与产品研究，不构成投资、法律或合规建议。

摘要

Polymarket 已经从加密原生预测市场，演化为事件概率定价基础设施的一类代表。它的核心价值不只是让用户交易事件结果，而是把政治、宏观、体育、科技、文化等不确定事件转化为可观察、可比较、可回测的概率信号。

原报告的大方向是成立的：预测市场正在主流化，Polymarket 具有流动性、品牌、数据和叙事优势；Agent 与预测市场之间也确实存在结合点。但原报告存在明显问题：部分交易量和活跃用户数据口径不可靠，把媒体报道或未来推演写成既成事实，对 API、收费、监管和 marketplace 冷启动的判断偏乐观。

本版核心结论：不要做“中文版 Polymarket”。更可行的路线是做一个面向 AI Agent 的概率与风险判断层：先自营预测数据、解释、回测和风险评分 API，再逐步扩展为 Web3 服务发现与 X402 按次付费平台。

一、事实校订

为了避免“看起来很像事实”的叙事误导读者，本节将原报告中的关键表述分为“需更正”“应降级为推测”和“基本准确”。

二、市场与业务判断

Polymarket 的护城河不是“链上”本身，而是流动性、品牌心智、市场覆盖、数据沉淀和媒体引用形成的复合网络效应。越多用户看 Polymarket 价格，越多交易者愿意在那里报价；越多交易者报价，价格越被媒体引用；越被引用，越像公共概率信号。

1. 为什么预测市场在 2024-2026 年突然主流化

  • 事件合约从小众工具进入新闻叙事。选举、利率、通胀、体育、战争、公司事件都适合被概率化表达。
  • 传统民调和专家预测的信任下降。用户更愿意看“用钱投票”的市场价格。
  • 加密基础设施成熟。稳定币、钱包、低成本链和链下撮合降低了交易摩擦。
  • 监管路径开始分叉。Kalshi 走合规交易所路径，Polymarket 则形成“国际加密原生 + 美国合规化”的双线结构。
  • AI 需要结构化概率输入。Agent 做规划、交易、研究和风控时，概率比新闻标题更容易进入决策模型。

2. Polymarket 的真实优势

优势 说明 是否可复制
流动性网络效应 热门市场的深度、价差和成交速度形成交易者惯性。 难，但不是不可复制；合规交易所和大型平台可通过补贴追赶。
品牌心智 “预测市场 = Polymarket”的媒体引用正在形成。 较难复制，需要热点事件长期积累。
市场创建与运营能力 事件标题、结算条件、品类运营、做市激励决定市场质量。 可学习，但细节密集。
数据资产 历史价格、成交、流动性、市场结果可用于概率校准和模型训练。 可被聚合，但原始交易数据和用户行为更有价值。
合规转型叙事 通过 Polymarket US 进入美国合规市场，提升机构可接受度。 可复制，但牌照、监管关系和执行成本高。

3. 商业模式应如何理解

Polymarket 的商业化不应只被理解为“交易手续费”。未来更有价值的收入层可能包括：

  • 交易费与流动性激励：部分市场 taker fee、maker rebates、taker rebates 共同调节流动性。
  • 数据分发：向金融终端、媒体、研究机构提供事件概率、历史数据和情绪指标。
  • 机构工具：风控、结算争议监控、合规报告、API 和白标服务。
  • 体育与娱乐合作：官方数据、品牌曝光、完整性监控和粉丝互动。

三、风险与监管

预测市场的核心风险不是单一监管文件，而是“交易、博彩、衍生品、消费者保护、市场诚信、广告合规、体育完整性”多条线并行。越主流，越会被监管以更细的粒度审视。

风险 具体表现 对创业者的启示
监管边界 政治、体育、战争、公共卫生等事件可能触碰不同监管红线。 信息服务、数据解释和工具层相对安全；撮合交易和代客交易风险显著上升。
结算争议 市场标题和 resolution criteria 若不清楚，会导致用户信任受损。 可做“结算规则风险评分”和“争议概率预警”。
市场操纵与内幕信息 小流动性市场容易被大户短期推动价格，体育和公司事件也可能有内幕交易。 异常交易检测、鲸鱼追踪、相关市场套利监控有商业价值。
安全风险 前端、钱包、签名、第三方服务供应链都可能成为攻击面。 Agent 调用交易或支付接口时必须有预算、白名单、权限和审计日志。
伦理与公共舆论 灾难、战争、名人生死等市场容易引发“用苦难赌博”的批评。 产品应支持事件黑名单、敏感品类隔离和解释性展示。
重要判断：“不直接做交易，只做信息服务”能降低风险，但不能自动等于“极低风险”。如果产品提供交易建议、自动执行、收益承诺、或面向受限地区用户规避访问限制，监管风险会迅速上升。

四、还应补充的深度

原报告覆盖面很广，但真正能拉开研究深度的不是更多叙事，而是对市场机制和产品需求的拆解。

1. 市场微观结构

应补充订单簿深度、价差、滑点、做市商激励、成交集中度、鲸鱼账户影响、相关市场套利等指标。预测市场价格不是天然等于真实概率，它还包含流动性成本、资金约束、情绪偏差和操纵噪声。

2. 概率质量评估

真正的“信息基础设施”必须可校准。建议用 Brier score、log loss、校准曲线、事件类别表现、时间衰减表现来评估不同预测源，而不是只看“某次大选猜对了”。

3. Agent 需求链路

Agent 不是天然会为概率买单。必须明确概率进入哪个工作流：交易决策、研究摘要、新闻解释、企业风控、政策监测、体育内容、还是个人助理提醒。不同工作流的频次、价格和准确性要求完全不同。

4. 合规分层

需要把产品能力拆成四层：只读数据、分析解释、交易建议、自动执行。每升一层，合规和安全要求都会显著增加。

六、AgentPort 可行性评估

AgentPort 的原始设想是：面向 AI Agent 的 Web3 API 服务聚合与发现平台，支持服务搜索、能力比较、按次调用，并通过 X402 实现自动化支付结算。这个方向有机会，但不宜一开始就做成“大而全的 Web3 API marketplace”。

7/10战略方向：Agent 支付与服务发现是早期但真实的基础设施机会。
5/10平台冷启动：双边 marketplace 难度高，早期应自营服务。
8/10预测切口：概率数据高频、低单价、可回测，适合展示 X402。
6/10商业化：微支付有叙事，但企业客户仍需要 SLA、发票和传统账户体系。

1. 为什么方向成立

  • Agent 需要外部能力。模型本身不能实时获取市场数据、链上状态、风险评分和交易执行能力。
  • Web3 API 碎片化。链上数据、风控、交易、预言机、预测市场、钱包和支付工具分散在不同供应商。
  • X402 适合机器支付。HTTP 402 + 稳定币支付让 API 按次调用、自动结算、无账号试用变得更自然。
  • 预测数据是好入口。它可验证、时效强、适合按次调用，也容易做出“服务质量比较”。

2. 为什么不能直接做大平台

  • 没有调用量，服务商不会来。没有服务商，Agent 开发者也不会来，这是典型双边冷启动困境。
  • API Key 不是最大痛点。开发者更关心稳定性、文档、响应质量、数据许可、SLA 和价格。
  • 微支付不是全部商业模式。0.001 美元一次适合演示，但高质量预测解释、风控、回测和交易前检查应该按价值定价。
  • 自动支付需要权限控制。Agent 不能无限制花钱，必须有预算上限、白名单、调用审计和异常暂停。
定位建议：AgentPort 早期不要讲“Web3 API 版 RapidAPI”，而要讲“Agent 可调用的概率与风险判断层”。先把一个高频、高价值、可评估的垂直能力做透，再把支付和发现能力平台化。

2. 核心 API

接口 能力 定价建议
/events/search 自然语言搜索事件、市场和相关问题。 免费或极低价，用于获客。
/probability/latest 返回某事件的多源概率、更新时间、成交量、流动性。 $0.001-$0.01/次。
/probability/explain 解释概率变化原因，引用新闻和市场变化。 $0.02-$0.20/次。
/market/risk-score 评估结算歧义、流动性不足、操纵风险。 $0.01-$0.10/次。
/backtest/source 按事件类别回测不同预测源的表现。 企业版或高价单次。
/alerts/create 创建概率变化、价差、异常成交提醒。 按月或按提醒量收费。

3. 数据质量评分

AgentPort 真正的壁垒不是把 API 列出来，而是告诉 Agent “哪个服务在什么场景下更可信”。建议评分体系包括：

  • Accuracy：Brier score、log loss、校准曲线。
  • Coverage：事件覆盖、语言覆盖、品类覆盖。
  • Freshness：更新时间、延迟、实时性。
  • Liquidity：成交量、价差、深度、滑点。
  • Resolution Risk：结算条件清晰度、争议历史、oracle 风险。
  • Cost：单次价格、批量价格、企业价格。

4. 支付策略

X402 是产品差异化，但早期不能只支持 X402。建议同时支持三种模式：

模式 适用对象 理由
X402 按次付费 Agent、黑客松、自动化脚本、Web3 开发者。 最能体现 Agent-native 和机器支付。
传统 API Key + 充值 普通开发者、小团队。 降低试用门槛，便于预算管理。
企业套餐 基金、媒体、研究机构、交易团队。 需要 SLA、权限、发票、审计和定制数据。

八、推广与运营方案

1. 目标用户分层

用户 痛点 触达方式

  • 公开榜单：每周发布 Best Prediction APIs by Accuracy，按品类比较预测源。
  • 事件专题页：美联储、选举、世界杯、Token TGE、宏观数据，每个事件都有概率、变化解释、API endpoint。

  • 服务商计划：等自营服务有真实调用量后，再邀请第三方模型、链上数据商、风控商上架。

Phase 0：垂直 MVP 4-6 周 Prediction API、MCP Server、x402 demo、10-20 个高质量事件页。 不开放 marketplace，不做自动交易。
Phase 1：质量比较 2-3 个月 多源概率比较、Brier score、结算风险评分、中文解释引擎。 不追求覆盖所有 Web3 API。
Phase 2：企业场景 3-6 个月 风控 dashboard、异常提醒、历史回测、团队权限和审计。 不承诺收益，不做代客交易。
Phase 3：平台开放 6-12 个月 第三方服务商 SDK、服务质量认证、分成结算、更多 Web3 数据品类。 不让未验证服务商直接进入推荐流。
最终判断：AgentPort 可行，但应先成为“Agent 可调用的概率与风险判断层”，再成为 Web3 服务发现平台。预测市场是入口，X402 是支付基础设施，长期护城河是数据质量、评估体系、工作流嵌入和被 Agent 反复调用的习惯。
